If you have read enough Python code, you have probably seen

__init__.py

all over the place.



Especially in big projects.



A lot of people know it has something to do with packages.



But they do not really know what it does, whether it is still

needed, or why imports start getting weird around it.



So let’s clear that up.



Years ago,

__init__.py

was used to mark a directory as a Python package.



In modern Python, that part is no longer required. A

directory with Python files can still work as a package without

it.



But that does not mean

__init__.py

stopped being useful.



It still does two important things.



First, it runs when the package is imported for the first

time, which makes it a good place for package-level setup.



Second, it lets you expose the main functions, classes, or

values you want people to import directly from the package,

instead of forcing them to import from deeper internal modules

every time.



That gives the package a cleaner public surface.



For example, let’s say you have a package called

utils,

and inside it you have

math_util.py.



From outside the package, this kind of import makes sense:



from utils.math_util import add



But inside

utils/__init__.py,

you do not want to write this:



from math_util import add



That often breaks.



Why?



Because once you are inside a package, Python needs to

resolve imports relative to that package context.



So inside

__init__.py

, you would usually write:



from .math_util import add



That dot matters.



It tells Python to look inside the current package.



This is where a lot of confusion starts.



People think the import line is wrong, when the real issue is

often that Python is resolving that line from a different context

than they expected.



That is also why the same code can work in one place and fail

in another.
