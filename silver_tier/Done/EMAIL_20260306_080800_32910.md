**<sub><sub>![P1 Badge](https://img.shields.io/badge/P1-orange?style=flat)</sub></sub>  Preserve mark events when batching outgoing audio**



Every audio chunk still increments `_mark_counter` and stores metadata in `_mark_data`, but `_buffer_flush_loop` and the interruption/end handlers call `_flush_outgoing_audio_buffer(None)`, which skips sending any corresponding mark. When multiple chunks are aggregated into one buffer, Twilio never receives the earlier mark IDs, so `_mark_data` entries for those chunks are never acknowledged or cleared and the playback tracker never learns that their bytes were played. In long calls this leaks mark metadata and leaves playback progress permanently behind. Consider emitting a mark for each stored chunk or collapsing the stored metadata to match the buffered payload before flushing.
