**Description:**

When using the OpenAI Agents SDK with **tracing enabled** (`set_tracing_export_api_key(...)`) and a **non-OpenAI model** (e.g., Gemini) for generation, `handoff` between agents fails with the following error:



```

openai.BadRequestError: Error code: 400 - {

    'error': {

        'message': "Invalid 'input[1].id': '__fake_id__'. Expected an ID that begins with 'fc'.",

        'type': 'invalid_request_error',

        'param': 'input[1].id',

        'code': 'invalid_value'

    }

}

```



---



**Expected Behavior:**



* Handoffs should work the same way as when tracing is **disabled**, even with non-OpenAI models.

* The tracing dashboard should either:



  * Gracefully handle the absence of an OpenAI `fc_` ID, **or**

  * Generate a synthetic trace ID to maintain the event graph.



---



**Actual Behavior:**



* When tracing is **disabled**, the handoff works fine with Gemini.

* When tracing is **enabled**, the SDK sends a placeholder `"__fake_id__"` to the OpenAI tracing API.

* The OpenAI API rejects it because it is not a valid `fc_` ID.



---



**Why This Matters:**



* Mixed-model workflows (e.g., using Gemini for generation + OpenAI tracing for workflow visibility) are currently **blocked**.

* Tools work fine in this setup — only handoffs break due to ID validation.

* This is inconsistent and limits multi-agent, multi-model orchestration.
