What exactly is Git Bisect?



Git bisect is a built-in command that uses binary search on your

commit history to find the exact commit that introduced a bug.



Instead of checking every commit by hand, Git will keep halving

the search space until it isolates the one bad commit.



If you have 200 commits to search, that means you only need to

test about 8 times (log₂(200) ≈ 8).



Here’s how it works...

----------------------



* Tell Git where the bug exists → mark a commit as bad.

​

* Tell Git a point in history where everything worked → mark as

good.

​

* Git will automatically checkout a commit halfway between.

​

* You test that commit (does the bug appear?).* If yes → git

bisect bad

​

* If no → git bisect good



* Git repeats the process until it finds the first bad commit.

​

* Run git bisect reset to return to your original branch.



A practical example

-------------------



Imagine your tests passed on commit a1b2c3 last week, but

today’s HEAD commit is broken.



# Start bisect

git bisect start



# Mark the broken commit (usually HEAD)

git bisect bad HEAD



# Mark a known good commit

git bisect good a1b2c3



Git now checks out a commit halfway in the search space (not

strictly by time, but in the commit graph).



You run your tests (or manually check the bug). Then tell Git:



git bisect good # if it works

git bisect bad # if it’s broken

git bisect skip # if the commit can’t be tested (e.g. it

doesn’t build)
