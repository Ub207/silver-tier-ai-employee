Hey there,



You’ve probably used __init__ before without really thinking

about it.



But did you know Python has a bunch of other hidden methods that

quietly control how your code behaves?



They’re called dunder methods, short for “double underscore.”



And once you understand them, you can make your own classes

behave just like the built-in ones...



So when you use something simple like a + b, Python doesn’t have

a special rule for that operator.



It just calls a.__add__(b) in the background.



Same with len(x), which calls x.__len__().



Even print(obj) is just asking for obj.__str__().



That’s what’s really happening inside Python.



It looks clean and simple on the surface because every class

already tells Python exactly how it should behave.



Here’s a quick example:



class InventoryItem:

def __init__(self, name, quantity):

self.name = name

self.quantity = quantity



def __add__(self, other):

if not isinstance(other, InventoryItem):

return NotImplemented

if self.name != other.name:

raise ValueError("Items must match")

return InventoryItem(self.name, self.quantity +

other.quantity)



def __repr__(self):

return f"{self.name}({self.quantity})"



Now try this:



apple1 = InventoryItem("Apple", 5)

apple2 = InventoryItem("Apple", 7)

print(apple1 + apple2) # Apple(12)



Basically, your objects learned to add themselves like numbers

because you defined the rule for it.



That’s a simple case, but once you start thinking this way,

you’ll realize you can design objects that behave like real

numeric types.
