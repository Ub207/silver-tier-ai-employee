Thanks for the feedback! You're absolutely right that the current error message is clear. If you prefer to keep the existing behavior, that's totally fine.



The main reasons I suggested this change:



1. **Exception semantics**: `AssertionError` typically indicates programming logic errors (should never happen), while `UserError` indicates API usage errors (can happen in normal use). Operations on a closed session fall into the latter category.



2. **Consistency with codebase**: Other API misuse errors in the realtime module use `UserError` (e.g., missing API key, unsupported tools, invalid handoff results).



3. **Reliability**: Assertions can be disabled with `python -O`, though this is rare in production.



4. **Earlier failure**: Fails at the session layer (where `_closed` flag lives) rather than propagating to the model layer.



However, these are minor improvements. If you feel the current behavior is sufficient, I'm happy to close this PR. What do you think?
