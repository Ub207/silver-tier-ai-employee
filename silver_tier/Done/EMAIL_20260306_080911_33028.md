Jash271 left a comment (openai/openai-agents-python#1938)



Thanks for the feedback! I just wanted to add that this would actually be a very common pattern when persisting sessions.



The core issue is that if you’re storing sessions in a database, you almost always need to associate them with something — a user ID, workspace, or tenant. Without that association, developers often end up maintaining a separate mapping table.



Common scenarios include:



- load all sessions for user_123

- Load conversations for workspace_456

- Display session titles in a history UI

- Clean up old sessions for a specific user



I do acknowledge that this might not be directly used by the Agent SDK itself, but it feels like a foundational building block that would make persistent session management much more practical for developers integrating the SDK.



Since we’re already creating database tables for sessions, adding basic metadata management feels like a natural extension rather than a highly specific feature. It’s completely opt-in and provides a standard pattern so users don’t have to build custom implementations themselves.



Happy to discuss further if there’s something I might be missing or not fully understanding!
