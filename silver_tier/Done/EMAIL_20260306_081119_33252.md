**<sub><sub>![P1 Badge](https://img.shields.io/badge/P1-orange?style=flat)</sub></sub>  File-backed sessions still skip shared lock**



`_lock_context()` returns `nullcontext()` whenever the database is file-based, so `get_items`, `add_items`, `pop_item`, and `clear_session` continue to run without any shared `threading.Lock`. The commit message claims to replace per-call lock creation with a shared lock for file databases, but that never happens here; concurrent callers will still execute unsynchronized on separate connections and can hit `sqlite3.OperationalError: database is locked` or interleaving writes. Consider returning `self._lock` for both memory and file paths to actually serialize access.
