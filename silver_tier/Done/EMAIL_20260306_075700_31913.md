From what i can see, there is no class called `ContextWrapper` in the codebase. The actual classes are:



1. `RunContextWrapper[TContext]`, The base context wrapper class  

2. `ToolContext[TContext]`, A subclass that extends `RunContextWrapper`



## What Actually Happens

`ToolContext` is not replacing or avoiding a standard Python dataclass. Instead:



1. **`ToolContext` *is* a dataclass**

2. **`ToolContext` extends `RunContextWrapper`**, It inherits from `RunContextWrapper` and adds additional tool-specific fields:

   - `tool_name`: The name of the tool being invoked  

   - `tool_call_id`: The ID of the tool call  

   - `tool_arguments`: The raw arguments string of the tool call  



The system works as lik this:



- When the agent runtime executes a tool call, it creates a `ToolContext` instance by calling `ToolContext.from_agent_context()`

- This method takes the existing `RunContextWrapper` and adds tool-specific information to it

- Functions that accept a context parameter will receive the appropriate context type:

  - **Regular context-aware functions** → `RunContextWrapper`  

  - **Tool functions** → `ToolContext` (includes all `RunContextWrapper` fields plus tool-specific fields)



## Why This Design?

This inheritance approach provides:



- **Enhanced functionality**: Tool functions get access to both the user's context and tool-specific information  

- **Type safety**: Different context types for different use cases  

- **Backward compatibility**: Code expecting `RunContextWrapper` still works with `ToolContext`  

- **Consistency**: Both classes are proper Python dataclasses with the `@dataclass` decorator  



**In short:** `ToolContext` *is* a standard dataclass, it’s just a specialized one designed for tool execution contexts, rather than general run contexts.
