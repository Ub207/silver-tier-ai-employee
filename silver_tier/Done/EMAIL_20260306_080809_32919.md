## Summary



Fixes #1443 - Changes Agent validation to raise `UserError` instead of `TypeError` when users pass invalid types to Agent parameters.



This provides better error messages and aligns with the SDK&#39;s documented behavior and existing patterns throughout the codebase.



## Changes Made



- Updated all `TypeError` instances in `Agent.__post_init__` to `UserError`

- Updated existing tests in `test_agent_config.py` to expect `UserError` instead of `TypeError`

- Added specific test case `test_tools_type_validation_issue_1443` to validate the fix for issue #1443



## Problem



When users passed invalid types to Agent parameters (e.g., a string instead of a list for the `tools` parameter), the SDK would raise `TypeError` instead of the more semantic `UserError`. This made it harder for users to understand that they were making a configuration mistake.



For example:

```python

agent = Agent(

    name=&quot;TestAgent&quot;,

    instructions=&quot;Test agent&quot;,

    tools=&quot;predict_weather&quot;  # should be a list

)

```



**Before**: Raised `TypeError: Agent tools must be a list, got str`  

**After**: Raises `UserError: Agent tools must be a list, got str`



## Solution



Changed all validation errors in `Agent.__post_init__` to raise `UserError` instead of `TypeError`, which is consistent with:

- The SDK&#39;s documented behavior

- Other validation errors throughout the codebase (see `guardrail.py`, `handoffs.py`, `prompts.py`, etc.)

- The semantic meaning of user input errors vs. general type errors
