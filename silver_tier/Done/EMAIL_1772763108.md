Thanks for sharing this. If your use case does not require handoffs, I would recommend using the agents-as-tools design pattern with structured inputs and outputs instead. That way, you can take more granular control over what information is passed into each model call.



If using handoffs is a hard requirement, I don't think using input_type is the only way to optimize the behavior. input_type only defines the schema of the internal function call used to hand off to another agent on the next turn. So, if you want to better manage model behavior after a handoff, you should focus on refining the agent instructions (fka system prompt), the handoff description, and the caller agent's instructions that trigger the handoff.
