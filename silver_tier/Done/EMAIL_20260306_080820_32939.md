## Root Cause



You identified the problem perfectly! When TTS providers (like ElevenLabs) emit odd-length audio chunks in MP3 format, `np.frombuffer(..., dtype=np.int16)` fails because int16 requires 2-byte alignment.



The error occurs in `src/agents/voice/result.py` line 91:

```python

np_array = np.frombuffer(b"".join(buffer), dtype=np.int16)

# ValueError: buffer size must be a multiple of element size

```



## The Fix



PR #1928 adds zero-byte padding for odd-length buffers:



```python

def _transform_audio_buffer(

    self, buffer: list[bytes], output_dtype: npt.DTypeLike

) -> npt.NDArray[np.int16 | np.float32]:

    # Combine all chunks

    combined_buffer = b"".join(buffer)



    # Pad with a zero byte if the buffer length is odd

    # This is needed because np.frombuffer with dtype=np.int16 requires

    # the buffer size to be a multiple of 2 bytes

    if len(combined_buffer) % 2 != 0:

        combined_buffer += b"\x00"



    np_array = np.frombuffer(combined_buffer, dtype=np.int16)

    # ... rest of the function

```



## Why This Works



- The padding only adds a single zero byte when needed

- Impact on audio quality is **negligible** (< 1 sample at 8kHz = 0.125ms of silence)

- Your exact workaround is now part of the official fix! 🎉



## Your Code Will Now Work



No more need for manual padding in your ElevenLabs model:



```python

# Before: You had to do this manually

if len(cur_chunk) % 2 != 0:

    cur_chunk += b"\x00"  

yield cur_chunk



# After: The SDK handles it automatically!

yield cur_chunk  # ✅ Works even with odd length

```



This will be available in the next SDK release. Thank you for the detailed reproduction case and for discovering the solution yourself!
