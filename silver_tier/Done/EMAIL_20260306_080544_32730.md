> Thanks for sending this, but I am not sure if having lots of `cast` is a good idea here.



You are right about avoiding many casts. The PR is updated to remove all per item casts.

It now accumulates plain dicts and applies a single cast at the return so the function still matches its annotated type while behavior stays identical. A short comment explains the Conversations API shape guarantee.

The only remaining suppression is a focused type ignore on item_id in pop_item because the union type does not guarantee that key, even though the API provides it at runtime.



If you prefer zero casts, a small TypeGuard that validates the dumped shape before append can replace the return cast.

Happy to switch to that pattern if it fits the codebase style better.
