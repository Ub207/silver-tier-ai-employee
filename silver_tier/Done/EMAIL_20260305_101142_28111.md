Thanks for clarifying! The tool only accepts defined parameters, and the absence of input items when using an agent as a tool is by design. Exposing the internal state of the caller agent (like conversation history or other data) would act like a global object, which makes complex use cases harder to manage. Because of that, we don’t plan to change this behavior.



I know this may not be the perfect answer for you, but I’d encourage designing your tool so it can take in enough information from the main agent. If you need a summary of the ongoing conversation, you can add a parameter for that in the function tool and ask the LLM to generate a useful summary when calling it. If you’d like to go with this approach, you can set up a function tool that internally calls Runner: https://openai.github.io/openai-agents-python/tools/#customizing-tool-agents  



Hope this helps clarify things and it was helpful to you.
